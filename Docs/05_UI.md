## Scope

The UI subsystem covers all SwiftUI views and view-supporting components. Spans `Views/` (top-level screen-scope views), `Components/` (reusable UI fragments smaller than a screen), and `ViewModels/` (the connective tissue between Models and Views). UI integration with the editing subsystem is described here at the view level; operations and tools themselves live in `Services/` (see `Docs/04_EDITING.md`).

## ContentView (in `Views/`)

To be expanded. Root window layout. Three-pane split view: sidebar on the left (track list, segment list, waypoint list), main map view in the center, inspector panel on the right (per-point properties, segment properties, project metadata). Toolbar at the top runs the full window width with tool selection and operation buttons. Status bar at the bottom (optional in v1).

## MapView (in `Views/`)

To be expanded. `NSViewRepresentable` wrapping `WKWebView`. Configures the WebView with the content rule list, registers the bridge message handler, loads `WebResources/index.html`. Implementation details for the bridge protocol live in `Docs/02_MAP_AND_BRIDGE.md`; this section covers the SwiftUI-side view wiring. Resizing and view-bounds changes are forwarded to the WebView automatically; explicit redraw triggers are bridge messages, not SwiftUI state changes.

## Sidebar (in `Views/`)

To be expanded. Hierarchical list of project structure: Tracks (master tagged distinctly via a small label or icon ‚Äî D-014 says no different visual treatment in the *map view*, but the sidebar's role is precisely structural visibility, so the master tag belongs here). Each track expands to its segments. Each segment expands to its waypoints. Click a track or segment to select all its points (per D-014 selection access paths). Right-click for per-item context menus (rename, change color, delete, mark as master, etc.). Drag to reorder (optional in v1).

## Inspector (in `Views/`)

To be expanded. Right-pane view that updates with the current selection's content. Three modes:

- **Point inspector** ‚Äî when a single point is selected, shows editable lat/lon/elevation (no per-point timestamps in working state per D-012). Snap-to-Ground button. Promote-to-Waypoint button.
- **Segment inspector** ‚Äî when a segment is selected (sidebar click or single-segment selection), shows editable name, color (color picker via the curated palette + standard macOS color picker for custom), parent track relationship, segment-level statistics.
- **Project inspector** ‚Äî when nothing is selected, shows project metadata (name, creation date, basemap selection, color palette overrides if any).

## Stats panel (in `Views/`)

To be expanded. Per-segment or per-track statistics: total distance, gain/loss, average and max speed (when timestamps are available), gradient histogram. May live as a separate panel or as a tab in the Inspector. Layout decision deferred until the design is in front of the user during M8 implementation. Selection-aware: shows stats for the current selection, or for the whole project if no selection.

## Toolbar (in `Views/`)

To be expanded. Top of the window. Tool selector (Point, Hand, Lasso, Brushes 1-4, Waypoint Place) with active-tool indicator. Operation buttons that depend on selection state (Delete, Snap to Ground, Trim Track, Remove GPS Spikes, etc.). Basemap selector control. Project save/export controls (Save, Save As, Import GPX, Export GPX, Export KML).

## Components (in `Components/`)

To be expanded. Reusable UI fragments. Anticipated set:

- `SegmentRow` ‚Äî a single segment in the sidebar list, with color swatch, name, master/subsidiary tag.
- `WaypointMarker` ‚Äî a waypoint icon for the sidebar list.
- `ColorSwatch` ‚Äî small color square, used in segment rows and the palette editor.
- `BrushSizeSlider` ‚Äî when brush size becomes user-configurable (deferred per D-016), this component provides the UI.
- `IconPicker` ‚Äî popover for selecting a waypoint icon from the curated set.
- `TrackListItem` ‚Äî a track in the sidebar with its expand/collapse affordance.
- `ToolbarButton` ‚Äî standard styled button used throughout the toolbar.

The Views/Components distinction is documented in CONVENTIONS.md "File organization." A Component is a building block that could appear in multiple views; a View is a screen.

## ViewModels (in `ViewModels/`)

To be expanded. Primary view model is `SessionViewModel` ‚Äî owns the project state (the loaded `GPXSession`), publishes changes via SwiftUI's observation system, mediates between view interactions and operations in `Services/`. Tool-active state, current selection state, hover state, and basemap selection live here.

Secondary view models may emerge during M8: a `SidebarViewModel` to handle expand/collapse and selection routing; an `InspectorViewModel` to handle the multi-mode display logic. Whether these earn their own files or fold into `SessionViewModel` depends on size ‚Äî the small-files principle from CONVENTIONS.md leans toward splitting.
